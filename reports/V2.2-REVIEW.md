### 2.3 技术方案

**评审结果:** ✅ 可行，技术选型成熟

**优点:**
1. **ORM 选型合理** - SQLAlchemy 2.0 支持异步，生态成熟
2. **数据库灵活** - SQLite/PostgreSQL 双支持，适应不同场景
3. **迁移工具完善** - Alembic 是 SQLAlchemy 官方迁移工具
4. **调度器成熟** - APScheduler 是 Python 主流定时任务库

**技术风险点:**
1. ⚠️ **异步兼容性** - 需确保整个调用链都是异步的，避免混用同步/异步代码
2. ⚠️ **连接池配置** - 生产环境需合理配置连接池参数，避免连接泄漏
3. ⚠️ **事务管理** - 需明确事务边界，确保数据一致性

**实施建议:**
```python
# 数据库连接配置建议
DATABASE_CONFIG = {
    'postgresql': {
        'host': os.getenv('DB_HOST', 'localhost'),
        'port': int(os.getenv('DB_PORT', 5432)),
        'database': os.getenv('DB_NAME', 'amazon_selector'),
        'user': os.getenv('DB_USER', 'admin'),
        'password': os.getenv('DB_PASSWORD', ''),
        'pool_size': 10,
        'max_overflow': 20,
        'pool_timeout': 30,
        'pool_recycle': 3600,
        'echo': os.getenv('DB_ECHO', 'false').lower() == 'true',
    }
}

# 异步 Session 工厂
async def get_db_session():
    async with async_session_maker() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise
        finally:
            await session.close()
```

## 四、风险提醒

### 4.1 技术风险

⚠️ **异步代码陷阱**
- SQLAlchemy 2.0 异步支持需要全程使用异步 API
- 混用同步/异步代码会导致性能问题或死锁
- **建议:** 在阶段 1 进行 POC 验证，确保团队熟悉异步编程

⚠️ **数据库迁移风险**
- 从 CSV 迁移到数据库涉及数据转换
- 历史数据可能丢失或格式不一致
- **建议:** 先备份 CSV 数据，编写数据迁移脚本并充分测试

⚠️ **连接池配置**
- 默认配置可能不适合生产环境
- 连接泄漏会导致数据库连接耗尽
- **建议:** 实现连接池监控，设置合理的超时和回收策略

### 4.2 进度风险

⚠️ **工作量可能低估**
- 40 小时估算基于理想情况
- 实际开发中可能遇到技术难点
- **建议:** 预留 20% 缓冲时间，优先保证核心功能

⚠️ **测试时间不足**
- 当前计划中测试时间占比偏低
- 数据库相关测试需要更多时间
- **建议:** 增加测试时间，确保测试覆盖率>90%

### 4.3 运维风险

⚠️ **历史数据膨胀**
- 每天采集会产生大量历史数据
- 30 天后数据量可能达到 GB 级别
- **建议:** 实现自动清理策略，定期归档旧数据

⚠️ **备份恢复未验证**
- 有备份工具但未进行恢复演练
- 真正需要时可能无法恢复
- **建议:** 在阶段 4 进行备份恢复演练
